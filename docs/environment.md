# Environment Variables

## Local development

There is a single `.env` at the **repo root** that holds every variable for every
process. The per-package files are symlinks back to it:

```
.env                          # canonical
apps/web/.env          â†’ ../../.env
apps/web/.env.local    â†’ ../../.env
apps/agent/.env        â†’ ../../.env
```

So `bun run web`, `bun run worker`, `next dev`, drizzle-kit, and the setup
scripts all read the same file. Edit the root `.env`, no per-package
duplication. The symlinks are checked into the repo (the targets â€” the actual
secret-bearing files â€” remain `.gitignore`d). On a fresh checkout, run
`cp .env.example .env` and fill values in.

The tables below describe what each **service** needs at deploy time on Render
(via `render.yaml`). In dev, the union of all of them lives in the root `.env`.

## Self-hosted Forgejo (optional, local dev only)

Forgejo and MinIO are **not** part of the Render Blueprint. They run locally via `docker-compose.yml` for self-hosted git workflows. If you only use GitHub as your forge provider, you can skip this entirely.

The `docker-compose.yml` defines:
- **`forgejo`** â€” Forgejo server on port `3000`, backed by Postgres and MinIO
- **`minio`** â€” S3-compatible storage for Forgejo LFS, attachments, and avatars
- **`postgres`** â€” shared database (port `5433` to avoid conflicts)

Forgejo-specific environment variables (`FORGEJO__database__*`, `FORGEJO__server__*`, `FORGEJO__storage__*`) are set directly in `docker-compose.yml` and consumed by Forgejo's built-in env config parser. They are not used by the OpenForge application code.

To provision Forgejo for local dev, run `infrastructure/forgejo/setup.ts` after `docker compose up`. It creates admin/agent accounts and emits the `FORGEJO_AGENT_TOKEN` and `FORGEJO_OAUTH_CLIENT_*` values to add to your `.env`.

## Security

See [architecture.md](architecture.md#authentication) for the auth model and invite flow.

- Sandbox: Render Private Service, no public IP, bearer-token auth, path-jailed workspaces, session-scoped secrets.
- All inter-service communication runs over Render's private network.
- LLM API keys saved from **Settings > API Keys** are encrypted in Postgres (AES-256-GCM) using **`ENCRYPTION_KEY`**; the worker decrypts with the same key at runtime. Environment variables remain a supported fallback.
- **`CI_RUNNER_SECRET`** protects **`POST /api/ci/results`**. Configure your GitHub Actions workflow (or other caller) to send this value in the `x-ci-secret` header.

For production hardening: rotate `AUTH_SECRET`, `SANDBOX_SHARED_SECRET`, and **`CI_RUNNER_SECRET`** periodically, and configure branch protection rules on critical repositories.
