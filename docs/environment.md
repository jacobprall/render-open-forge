# Environment Variables

## Local development

There is a single `.env` at the **repo root** that holds every variable for every
process. The per-package files are symlinks back to it:

```
.env                          # canonical
apps/web/.env          → ../../.env
apps/web/.env.local    → ../../.env
apps/agent/.env        → ../../.env
```

So `bun run web`, `bun run worker`, `next dev`, drizzle-kit, and the setup
scripts all read the same file. Edit the root `.env`, no per-package
duplication. The symlinks are checked into the repo (the targets — the actual
secret-bearing files — remain `.gitignore`d). On a fresh checkout, run
`cp .env.example .env` and fill values in.

The tables below describe what each **service** needs at deploy time on Render
(via `render.yaml`). In dev, the union of all of them lives in the root `.env`.

## Web service (`apps/web`)

| Variable | Required | Notes |
|---|---|---|
| `DATABASE_URL` | Yes | Auto-wired by `render.yaml` |
| `REDIS_URL` | Yes | Auto-wired by `render.yaml` |
| `AUTH_SECRET` | Yes | NextAuth encryption key, auto-generated by `render.yaml` |
| `AUTH_URL` | No | Public URL of the web app (NextAuth uses this for callbacks) |
| `ADMIN_EMAIL` | Yes | Email for the auto-bootstrapped admin account |
| `ADMIN_PASSWORD` | Yes | Password for the auto-bootstrapped admin account |
| `FORGEJO_INTERNAL_URL` | Yes | Auto-wired (`http://forge-forgejo:3000`) |
| `FORGEJO_EXTERNAL_URL` | Yes | Public URL of your Forgejo instance |
| `FORGEJO_AGENT_TOKEN` | Yes | Agent service account token |
| `ANTHROPIC_API_KEY` | Yes* | *Provider key required for LLM access unless keys are stored in Postgres (see `ENCRYPTION_KEY`) |
| `OPENAI_API_KEY` | No | Enables OpenAI models when set as env fallback |
| `ENCRYPTION_KEY` | Yes† | †64 hex chars (`openssl rand -hex 32`), **same on `forge-web` and `forge-agent`** — encrypts UI-managed API keys at rest. Auto-generated on `forge-web` and linked into `forge-agent` via the Render blueprint |
| `RENDER_API_KEY` | Yes | Render API key, starts CI tasks on **`forge-ci`** |
| `RENDER_CI_WORKFLOW_SLUG` | No | Default **`forge-ci`**, must match your Workflow deployment slug |
| `CI_RUNNER_SECRET` | Yes | Auto-generated on web and synced to **`forge-ci`** via Blueprint |
| `CI_CALLBACK_URL` | No | Override callback base URL if tasks cannot POST to the default app URL |
| `CI_RUNNER_MODE` | No | Omit or **`render`** on Render. Use **`local`** only for laptop dev |

## Security

See [architecture.md](architecture.md#authentication) for the auth model and invite flow.

- Sandbox: Render Private Service, no public IP, bearer-token auth, path-jailed workspaces, session-scoped secrets.
- All inter-service communication runs over Render's private network.
- LLM API keys saved from **Settings > API Keys** are encrypted in Postgres (AES-256-GCM) using **`ENCRYPTION_KEY`**; the worker decrypts with the same key at runtime. Environment variables remain a supported fallback.
- **`CI_RUNNER_SECRET`** protects **`POST /api/ci/results`**. The Blueprint keeps web and **`forge-ci`** in sync.

For production hardening: rotate `AUTH_SECRET`, `SANDBOX_SHARED_SECRET`, `FORGEJO_AGENT_TOKEN`, and **`CI_RUNNER_SECRET`** periodically, and configure branch protection rules on critical repositories.
